DIAGRAMA ENTIDAD-RELACIÓN (DER)
Sistema: Inventario de Bienes
Fecha de actualización: 2026-03-06

1. Objetivo del diagrama
   - Representar entidades, atributos y relaciones según el estado actual
     del sistema (incluyendo permisos por usuario y catálogo de categorías).

2. Entidades

   2.1. Entidad USUARIO (users)
        - id (PK)
        - name
        - email
        - email_verified_at
        - password
        - role (admin / user)
        - permissions (JSON, opcional)
        - security_color_answer
        - security_animal_answer
        - security_padre_answer
        - login_attempts
        - locked_until (nullable)
        - remember_token
        - created_at
        - updated_at

   2.2. Entidad BIEN (bienes)
        - id (PK)
        - nombre
        - codigo (ÚNICO)
        - descripcion
        - categoria (texto, nullable)
        - ubicacion (nullable)
        - estado (bueno/regular/malo/de_baja)
        - fecha_adquisicion (nullable)
        - created_at
        - updated_at
        Nota: el campo valor fue eliminado del esquema actual.

   2.3. Entidad CATEGORIA (categorias)
        - id (PK)
        - nombre (ÚNICO, longitud 30)
        - estado (activo/inactivo)
        - created_at
        - updated_at

   2.4. Entidad BITÁCORA (bitacoras)
        - id (PK)
        - user_id (FK a users, nullable)
        - modulo
        - entidad_id (nullable, referencia lógica)
        - accion
        - resultado (default: exito)
        - descripcion (nullable)
        - created_at
        - updated_at

   2.5. Entidad PASSWORD_RESET_TOKEN (password_reset_tokens)
        - email (PK)
        - token
        - created_at

   2.6. Entidad SESSION (sessions)
        - id (PK)
        - user_id (nullable)
        - ip_address
        - user_agent
        - payload
        - last_activity

3. Relaciones entre entidades

   3.1. USUARIO — BITÁCORA
        - Tipo: 1 a N
        - Un usuario puede generar muchas entradas de bitácora.
        - Una bitácora puede quedar sin usuario si se elimina (nullOnDelete).

   3.2. USUARIO — SESSION
        - Tipo: 1 a N (lógica)
        - Un usuario puede tener varias sesiones.

   3.3. BIEN — CATEGORIA
        - Tipo: N a 1 (lógica de catálogo)
        - bienes.categoria guarda el nombre; categorias.nombre mantiene catálogo.
        - No hay FK estricta entre ambas tablas (sincronización por aplicación).

   3.4. BITÁCORA — ENTIDAD AFECTADA (relación polimórfica lógica)
        - bitacoras.modulo + bitacoras.entidad_id identifican el registro afectado
          (ej.: bienes, usuarios, auth, categorías).

4. Notas de diseño vigentes

   - Permisos granulares por usuario:
     * Se almacenan en users.permissions (JSON).
     * Si role=admin, el sistema concede acceso completo automáticamente.
     * Si role=user, se aplican permisos seleccionados en creación/edición.

   - Reglas de seguridad de cuentas:
     * Bloqueo tras intentos fallidos (login_attempts, locked_until).
     * Desbloqueo controlado por administrador.

   - Continuidad administrativa:
     * El sistema evita eliminar o degradar al último administrador.
