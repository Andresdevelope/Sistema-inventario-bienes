DIAGRAMA DE FLUJO DE DATOS (DFD)
Sistema: Inventario de Bienes
Fecha de actualización: 2026-03-06

1. Nivel 0 (Contexto)

   Entidades externas:
   - Usuario administrador
   - Usuario operador
   - Servicio externo reCAPTCHA (Google)

   Proceso principal:
   - P0: Sistema de Inventario de Bienes

   Almacenes de datos:
   - D1: BD_Usuarios (users)
   - D2: BD_Bienes (bienes)
   - D3: BD_Categorías (categorias)
   - D4: BD_Bitácora (bitacoras)
   - D5: BD_Sesiones (sessions)

   Flujos principales:
   - E1: Credenciales y token reCAPTCHA -> P0
   - S1: Resultado de autenticación -> Usuario
   - E2: Solicitudes bienes/categorías/reportes -> P0
   - S2: Respuestas de inventario -> Usuario
   - E3: Solicitudes gestión de usuarios/permisos (solo admin) -> P0
   - S3: Respuesta gestión de usuarios/permisos -> Usuario admin
   - E4: Consultas de bitácora -> P0
   - S4: Resultados de auditoría -> Usuario admin

2. Nivel 1 (Descomposición)

   P1: Autenticación y control de acceso
       Entradas:
       - Usuario + contraseña
       - Token reCAPTCHA
       Salidas:
       - Acceso concedido/denegado
       - Mensaje de bloqueo si corresponde
       Acceso a datos:
       - Lee D1 (validar usuario/rol/locked_until)
       - Escribe D4 (eventos login/logout/bloqueo)
       - Escribe D5 (sesiones)

   P2: Gestión de bienes y categorías
       Entradas:
       - Alta/edición/baja/consulta de bienes
       - Gestión de categorías
       Salidas:
       - Confirmación de operación
       - Listados y reportes
       Acceso a datos:
       - Lee/escribe D2 (bienes)
       - Lee/escribe D3 (categorías)
       - Escribe D4 (traza de acciones)

   P3: Gestión de usuarios y permisos
       Entradas:
       - Crear/editar/eliminar/desbloquear usuarios
       - Asignación de rol y permisos
       Salidas:
       - Confirmación/errores de validación
       Acceso a datos:
       - Lee/escribe D1 (incluye users.permissions)
       - Escribe D4 (acciones sobre usuarios)
       Reglas clave:
       - Solo admin asigna rol admin.
       - No se elimina/degrada el último admin.

   P4: Auditoría (bitácora)
       Entradas:
       - Filtros de consulta de bitácora
       Salidas:
       - Listado de eventos
       Acceso a datos:
       - Lee D4

3. Flujo específico nuevo: alta de usuario con permisos

   1) Admin abre modal de crear usuario.
   2) Frontend aplica límites y sanitización básica:
      - name max 30
      - email max 60
      - password max 25
      - respuestas de seguridad max 30
   3) Backend sanitiza y valida con reglas robustas (regex + anti HTML).
   4) Si role=admin, el sistema asigna permisos completos automáticos.
   5) Si role=user, guarda permisos seleccionados en users.permissions.
   6) Registra evento en bitácora y retorna respuesta al admin.
